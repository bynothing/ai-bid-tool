对,你点出了这套工具真正的核心——**不在渲染,在决策**。模板和各种渲染器只是"能力",真正难、也真正决定成败的,是在每一张图上有效地选出最合适的产出路径,并且在最理想的路径走不通时优雅降级、绝不硬失败。把这个补进来后,决策层的输出就不再是"选哪个渲染器",而是"**走哪一级产出路径**",形成一个按"设计可控性"递减、"通用兜底性"递增的三级降级链:三级各自靠什么保证、为什么这样排序:

**Tier 1(首选)SVG 模板 + 契约生成。** 设计可控性最高,但要求内容能落进某个模板的契约。决策层先试这一级:LLM 读能力目录里的模板摘要选模板,取契约,按契约生成,数据级校验通过即出图。这是产出精美图的主路。

**Tier 2(降级)柔性渲染器。** 当没有任何 SVG 模板的契约能装下这份内容(变体都试过、拆分也不行)时降到这一级。它牺牲一部分设计精致度,换来对任意内容的鲁棒性——而鲁棒性恰恰来自这些渲染器**自带的安全网**:echarts 数据驱动、随数据量自适应;mermaid 文法合法即出图、不会摆崩;graphviz 自动布线、节点再多也能排开;html_css 浏览器自动版式。决策层在这一级按内容信号选其中之一,LLM 也可参与判断"这份内容用哪个柔性渲染器最合适"。

**Tier 3(兜底)自由矢量 SVG。** 最后手段,设计与稳定保证最低,但**保证一定能出图**。它的存在只是为了让系统永不硬失败——任何走到这里的内容,质量靠不住但不会开天窗。所以它排在最底,只在前两级都不可用时才触发,且应该在 manifest 里明确标记"兜底产出,建议人工复核"。

这条链的关键性质是:**决策层永远从最可控的一级开始往下试,只在上一级确实不可行时才降级,且每一步降级都有据可查。** 这样既保住了"大多数图精美稳定"(走 Tier 1),又保证了"任何图都出得来"(最差有 Tier 3)。

## 这对决策层和计划的影响

决策层从一个普通模块升格为**重构的一等核心**,它要消费一份新的产物——**工具能力目录**:登记当前所有 SVG 模板(摘要 + 契约),以及各柔性渲染器的能力描述(各自擅长/不擅长什么、输入要求)。LLM 选路径、软件做降级判断,都依赖这份目录。它和模板注册表共址,但范围更大(模板 + 渲染器一起)。

并进原计划的改动是几处增量,不推翻前面的阶段:

**Phase 0** 增一项:把 `core/decision.py` + `core/router.py` 明确定位为"核心决策层",并定义"工具能力目录"的数据结构(模板项 + 渲染器能力项)。`api.plan()` 的返回也要能表达"走了哪一级、为什么降级"(在 `decision` 里加 `tier` 与 `degraded_from`)。

**Phase 2** 把原来的回退阶梯扩成**完整的三级降级链**,这是本轮新增的核心:Tier 1 内部先走变体阶梯(换更大变体 → 拆成两图 → 通用保底模板),整层不可行再降到 Tier 2(按内容信号 + LLM 判断选柔性渲染器),Tier 2 仍不合适才到 Tier 3 自由 SVG。每一步降级都写进 manifest,Tier 3 强制标记需人工复核。决策层据能力目录驱动这整条链。

**Phase 3** 扩类时多一条好处:新图型即使还没来得及做 SVG 模板,也能直接靠 Tier 2/3 出图,**不阻塞流水线**——有模板就精美,没模板也不开天窗。这让"逐类补模板"可以从容推进,而不是非得全部就位才能上线。

合并后的完整降级链可以记成一句话:**变体阶梯 → 拆分 → 保底模板(以上属 Tier 1)→ 柔性渲染器(Tier 2)→ 自由矢量 SVG(Tier 3)**,决策层基于 LLM + 能力目录从上往下逐级尝试,首选精美、保底不失败,全程留痕。
