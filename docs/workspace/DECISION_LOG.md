# Decision Log

本文件记录长期有效的架构决策。每条决策应说明背景、结论、原因、影响和后续动作。

## D-001：项目记忆不依赖智能体长期记忆

日期：2026-06-07

状态：采用

背景：

`bid-tool` 是一个持续开发的复杂项目，配图能力涉及流水线、Schema、LLM prompt、模板、渲染器、QA、Word 装配等多个层面。任何一次新会话如果依赖智能体记忆，都会出现上下文漂移和重复分析。

决策：

建立 `docs/workspace/` 项目记忆系统，并将 `WORKSPACE.md` 作为固定入口。

原因：

- 文件可搜索、可审查、可版本化。
- 新会话可通过少量固定文件恢复上下文。
- 状态、计划、背景知识、决策记录分离，减少巨型文档失控。

影响：

- 每次重要开发结束必须更新 `STATE.md` 或 `SESSION_HANDOFF.md`。
- 架构取舍必须进入 `DECISION_LOG.md`。
- 任务状态变化必须进入 `ROADMAP.md`。

## D-002：自动配图主路线采用“LLM 语义决策 + 确定性渲染”

日期：2026-06-07

状态：采用

背景：

标书配图要求稳定、正式、可审计、可批量生成。让 LLM 直接写 SVG 坐标成本高、质量波动大、难以保证文字和几何稳定。

决策：

主路线采用：

```text
章节内容
  -> LLM 选择图型与模板
  -> LLM 填充结构化 Job
  -> 代码校验契约与容量
  -> 代码渲染 SVG/PNG
  -> QA 与 manifest 留痕
```

原因：

- LLM 擅长语义理解和内容归纳。
- 程序擅长约束校验、排版、渲染和重复执行。
- 结构化 Job 便于测试、缓存、复用和人工审查。

影响：

- 新图型应优先沉淀模板契约和 fixture。
- Prompt 不应要求模型输出 SVG 坐标。
- 渲染器必须暴露可验证的失败信息。

## D-003：配图产出路径采用 Tier 降级链

日期：2026-06-07

状态：采用

背景：

只靠模板会遇到覆盖不足，只靠自由渲染会损失质量和稳定性。系统需要既追求高质量，又保证不开天窗。

决策：

采用三级产出路径：

| Tier | 路线 | 目的 |
| --- | --- | --- |
| Tier 1 | 模板 + 契约生成 | 高质量、高稳定、低成本 |
| Tier 2 | 柔性渲染器 | 覆盖模板暂未支持的内容 |
| Tier 3 | 自由矢量兜底 | 保证一定有图，但强制人工复核 |

降级链：

```text
模板变体
  -> 拆分
  -> 保底模板
  -> 柔性渲染器
  -> 自由矢量兜底
```

原因：

- Tier 1 质量可控，是长期主路线。
- Tier 2 保证通用性，适合数据图、复杂拓扑、标准流程。
- Tier 3 防止流水线硬失败，但不能被当作合格终点。

影响：

- `api.plan()` 和 manifest 必须记录 tier 与降级原因。
- fallback 必须可审计，不能静默发生。
- 高频 Tier 3 场景应转化为模板建设需求。

## D-004：真实样本优先于示例样本

日期：2026-06-07

状态：采用

背景：

示例 Job 容易服务于已有实现，不能代表真实标书文本复杂度、术语密度和评审表达要求。

决策：

后续质量评估以真实或近真实标书章节片段为主要 fixture。

原因：

- 配图工具的价值来自真实投标场景。
- 真实样本能暴露节点密度、文本长度、图型选择、Word 可读性等问题。
- fixture 能把主观质量问题转化为可复现工程任务。

影响：

- 每个重要缺陷应优先沉淀为 `tests/fixtures/illustration_cases/`。
- 质量评估不能只看“是否生成文件”。

## D-005：项目级文档统一放入 docs

日期：2026-06-07

状态：采用

背景：

历史上部分长期架构、重构计划和绘图标准文档放在 `src/bid_tool/` 内部，导致源码目录同时承担运行时代码和项目知识库两种职责。随着项目记忆系统建立，文档需要按用途集中，方便新会话快速恢复上下文，也避免源码包目录继续膨胀。

决策：

将项目级文档统一放入 `docs/`：

- 工作状态、计划、决策和交接放入 `docs/workspace/`。
- 长期架构和系统工程计划放入 `docs/architecture/`。
- 绘图内容标准和质量标准放入 `docs/standards/`。

源码目录 `src/bid_tool/` 只保留运行时代码、Schema、Prompt、模板契约、能力目录、包内示例和运行时素材。

原因：

- `docs/` 更适合作为可维护、可检索、可版本化的项目知识库。
- `src/` 保持运行时边界清晰，减少“文档即源码资源”的混淆。
- 新增文件有明确归位规则，降低后续结构熵增。

影响：

- 新增长期方案文档不再放入 `src/bid_tool/`。
- 配图模板自己的 `README.md`、`contract.json`、`analysis.md` 仍留在 `src/bid_tool/illustration_v2/templates/`，因为它们是模板能力的一部分。
- 文件归位以 `docs/PROJECT_STRUCTURE.md` 为准。

## D-006：配图质量改造采用人工目标驱动

日期：2026-06-07

状态：采用

背景：

当前系统已经具备配图能力，但产出效果不稳定：图不够美观、信息表达不清楚、箭头混乱，尚未达到工程级标书图的交付标准。此类问题如果只靠开发者主观调样式，很容易变成零散修补，无法持续提高。

决策：

建立 `docs/quality-targets/illustration/` 作为配图质量目标区。后续配图质量优化必须先对齐人工参考目标、评分标准和当前输出基线，再进入具体开发。

质量改造链路：

```text
人工参考图
  -> 目标说明
  -> 当前输出评分
  -> 缺陷记录
  -> Todo
  -> fixture / QA
  -> 渲染器或模板改造
  -> 再评分
```

原因：

- “美观”和“清楚”必须转化为可讨论、可评分、可回归的工程指标。
- 箭头混乱、信息不清楚等问题需要样本驱动，而不是只调颜色和样式。
- 人工参考目标能为模板、布局、箭头策略、质量门禁提供稳定方向。

影响：

- 新的配图质量任务优先写入 `docs/quality-targets/illustration/TODO.md`。
- 参考图和人工评审记录放入 `references/` 与 `reviews/`。
- 当前系统输出基线放入 `generated-baselines/` 或转为 `tests/fixtures/illustration_cases/`。
- 质量优化完成后必须记录在 `RECORD.md`，并尽量沉淀为 fixture 或 QA 规则。

## D-007：工艺流程 / 系统交互大图先走 Draw.io Tier 2 路径

日期：2026-06-08

状态：采用

背景：

参考图“总装 A-H 工艺段流程及系统交互点分析图”不是普通线性流程图，而是多工艺段分区、段内步骤、主流程、辅助/回流关系、系统交互点、图例和术语说明的复合工程图。当前尚无冻结 Tier 1 模板可直接承载此结构。

决策：

新增 `process.interaction_map` 图型。决策层遇到该图型或具备 `sections + primary_flow/support_flows` 的结构化 Job 时，优先路由到 Draw.io renderer，作为 Tier 2 首个实现路径，并标记 `needs_human_review=true`。

原因：

- Draw.io 适合先验证复杂分区、可编辑源文件和正交连接线。
- 该图型仍需人工目标反复校准，过早冻结 Tier 1 模板会增加返工成本。
- 当前阶段更需要可复现基线、评分和缺陷清单。

影响：

- `process.interaction_map` 进入 `AI_ILLUSTRATION_API_V2.md`。
- 回归样本进入 `tests/fixtures/illustration_cases/process_interaction_map/`。
- 后续若该图型评分稳定达到 80+，应将高频结构沉淀为 Tier 1 模板候选。
